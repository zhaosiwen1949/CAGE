# eval_floorplan.py — 用真值户型评估预测结果

把 `align_floorplan.py` 输出的 `{name}_aligned_polys.json` 与 realsee 真值户型
（如 `data/custom/xinghewan_floorplan/`）自动配准后对比，量化房间几何、门/窗位置的
准确性。无 torch 依赖，本机可实跑。

## 用法

```bash
conda activate CAGE
bash tools/eval_floorplan.sh \
    infer_out/xinghewan_da3_mvs_aligned_polys.json \
    data/custom/xinghewan_floorplan \
    infer_out
```

产出：
| 文件 | 内容 |
| --- | --- |
| `{name}_eval.json` | 配准参数 + 全部指标 + 逐房间/逐 opening/逐门窗明细（机器可读全量） |
| `{name}_eval.txt` | 控制台打印的指标摘要原文（比 json 简洁直观，含门窗位置段 + 合并 GT 再评轮） |
| `{name}_eval_overlay.png` | 房间面板（GT 蓝描边叠预测橙填充、标 IoU）+ 有 doors_windows.json 时追加门/窗位置面板 |

## 基准数据格式（gt_dir）

- `room_layout.json`（必需）：realsee 逐 pano 房间墙线。米制、y-up、全局统一坐标系；
  每个 pano 里**等高（y 不变）的墙线**投影到 (x,z) 平面即房间多边形。同名房间多 pano
  取并集。个别房间（卫C）的环不闭合（缺一段小台阶边），加载器会自动补线。
- `rooms_extra.json`（可选）：`room_layout.json` 缺失房间的补充（米制多边形），加载时自动合并。
  从页面户型图 SVG 恢复（见下"从页面 SVG 恢复缺失房间"）。文件路径由 `floorplan.json`
  顶层 `rooms_extra.local_path` 字段指定，缺省回退同目录 `rooms_extra.json`。
- `rooms_centerline.json`（可选，**默认基准**）：全部房间的**墙体中线**多边形（米制，
  room_layout 世界系）。SVG 户型图按墙中线绘制，配准后原样输出（不做内缩）即得。
  与预测的零厚共享墙**同口径**，消除内表面基准的半墙厚系统差（见下"天花板效应"）。
  路径由 `floorplan.json` 顶层 `rooms_centerline.local_path` 指定；文件不存在时
  自动回退内表面基准（`--gt_geometry inner` 可强制）。
- `doors_windows.json`（可选）：门/窗/门洞的**位置**清单，门窗评估的唯一 GT 口径
  （旧的 `openings_gt.json` 连通清单已弃用、不再读取）。从户型图 base SVG 恢复、投影到墙中线，
  米制/room_layout 世界系。格式：
  ```json
  {"items": [{"type": "door", "subtype": "单开门", "room": "卧室A",
              "center": [x, z], "segment": [[x1,z1],[x2,z2]], "width_m": 0.81}]}
  ```
  `type` ∈ door/window/opening（门/窗/门洞）；`segment` 为门窗占据的墙中线子段，`center` 为其中点。
  路径由 `floorplan.json` 顶层 `doors_windows.local_path` 指定，缺省回退同目录 `doors_windows.json`；
  文件不存在则跳过门窗位置评估（见"指标解读 §4"）。
  ⚠️ 不能从 `room_layout.json` 的 `state=false` 自动提取门窗：实测该字段既含真门，
  也含家具遮挡的实墙和外墙窗（客厅↔卧室B 之间 3.45m 实墙也标 false），只能人工/SVG 标注。

## 流程

1. **载入 GT**：默认读 `rooms_centerline.json`（墙中线，`--gt_geometry` 控制）；
   inner 口径则逐 pano 水平墙线 → `shapely.polygonize` → 同名房间并集 → `simplify(0.03)`。
2. **载入预测**：`rooms[].pixel` + `normalization` 按 `pixel_to_world` 公式还原米制坐标。
   **每轴独立仿射**（x/y 跨度不同），不能等比缩放；`hflip` 时列坐标按 `255-col` 解码。
   `--pred_units m|mm`（默认 m）指定 .ply 原始单位——**不要信任 `world_mm` 字段名**，
   MVS 场景它存的其实是米。
3. **自动配准**：两边都已曼哈顿对齐，只搜 4×90° 旋转 × 是否镜像 × 尺度网格
   （默认 0.90–1.10 步 0.01）；每个候选用栅格化 union mask 的互相关
   （`cv2.matchTemplate`，5cm/px 粗搜 + 1cm/px 精修）求最优平移，以 union-IoU 取全局最优。
   **拟合出的 scale 本身是点云尺度质量指标**（星河湾 ≈0.98，即 MVS 尺度偏大 ~2%）。
4. **房间匹配**：IoU 矩阵贪心 1:1 匹配（`--match_iou 0.5`）。
5. **指标**（见下）。
6. **合并 GT 再评一轮**：若某预测房把多个 GT 房整个吞并（每间被覆盖 ≥95%），把这些
   GT 房 union 成一间后重算全部房间/角点/墙线指标（见下"合并 GT 再评一轮"）。

## 指标解读

指标分五组：**配准质量 → 房间几何 → 角点/墙线几何 → 门/窗位置 → 合并 GT 再评**。
下面每个术语先给一句大白话，再说这个场景里怎么读。括号内数字是星河湾当前实测值
（21 房完整 GT，**默认中线口径**）；门/窗位置一组星河湾无标注、跳过，示例数字取自 huizhongbeili-106。

### 0. 先看配准（registration）

配准 = 把预测户型旋转/镜像/缩放/平移，套到 GT 上对齐。控制台第一行：
`rot 180 deg, mirror False, scale 0.990, IoU 0.963`。

- **scale（尺度）**：把预测放大/缩小多少倍才与 GT 面积吻合。0.990 = 预测比真实**大 1%**，
  即 MVS 重建的点云尺度偏大约 1%——**这个数本身就是点云尺度质量的体检结果**，越接近 1 越好。
  （inner 口径下拟合出 0.980：GT 内表面比中线小一圈，会把 scale 额外压低 ~1%，
  中线口径的 0.990 才是点云尺度的真实读数。）
- **union IoU（总体交并比）**：见下 IoU 定义，这里是**两边所有房间合成一整块**后的重叠度。
  0.963 是全局对得有多准的总分；配准明显失败时这个数会掉到 0.5 以下。

### 1. 房间级（room-level）

一句话：预测的每个房间，形状和位置和真实房间有多像。

- **IoU（Intersection over Union，交并比）**：两个多边形**重叠面积 ÷ 合并面积**。
  完全重合=1，完全不沾=0。这是判断"两个房间算不算同一个"的核心尺子。
  逐房列出（如 `厨房 0.924`、`卧室B 0.903`），越大越准。
- **mean IoU（平均交并比）**：所有**成功配上对**的房间 IoU 的平均（0.849）。反映匹配上的
  房间整体贴合程度。注意它只对匹配房间取平均，不含漏检/合并的房间。
- **房间匹配（matching）**：按 IoU 从高到低，把预测房和 GT 房**一对一**配对，IoU 要 ≥
  `--match_iou`（默认 0.5）才算配上。配上的叫 matched（14 个）。
- **precision / recall / F1（精确率 / 召回率 / F1 值）**——判断"房间数量对不对"的三个分数，
  都在 0~1 之间：
  - **precision（精确率）= 配对成功数 ÷ 预测房总数**（14/17=0.824）：预测出来的房间里，
    有多大比例是真的（没有凭空多切、错切）。
  - **recall（召回率）= 配对成功数 ÷ GT 房总数**（14/21=0.667）：真实存在的房间里，
    有多大比例被预测覆盖到了（没漏、没被并掉）。
  - **F1 = precision 和 recall 的调和平均**（0.737）：两者的综合分，任一方低都会把它拉低。
- **未匹配的诊断**（关键：区分"真错"和"合并/分割"）：
  - **未匹配 GT 房 → 合并诊断**：`客厅 98% covered by p15 (likely merged)` 意思是"客厅"这个
    真实房间几乎整个落在预测房 p15 里——**不是漏检，是被并进了更大的房间**。星河湾 7 个未匹配
    GT 房全是这种：客厅/餐厅/阳台B 三间被合成一个大 p15；卫D/过道D 被切得和 GT 对不齐；
    阳台A 并入卧室A、阳台C 并入卧室C。
  - **未匹配预测房 → 成分诊断**：`p15 covers 客厅 47% + 餐厅 38% + 阳台B 14%` 把这个"多出来的"
    预测房拆解成它盖住了哪几个 GT 房，一眼看出它是几个房间合并的产物。
  - 所以 recall 0.667 偏低**几乎全是房间合并/分割**造成的，而非模型把墙画飞了——这是当前
    预测链路最值得改进的点。量化"接受合并后有多准"见下第 5 组。

### 2. 角点几何（corners）

一句话：预测的墙角，和真实墙角在位置上差多少厘米。

- **corner precision/recall @ 阈值**：把预测房的每个角点和 GT 角点比，距离 ≤ 阈值就算"命中"。
  阈值取多档（`--corner_tol` 默认 0.1 / 0.2 / 0.3m）：
  - `@0.3m P 0.870 R 0.779`、`@0.2m P 0.749` = 允许 20~30cm 误差时，多数预测角点能对上。
  - `@0.1m P 0.160` = 收紧到 10cm，命中率骤降——**这不代表墙画歪了**：预测坐标来自
    256×256 密度图，一个像素就约 10cm（24.9m/255≈9.8cm），角点位置天生带 ~10cm 量化格；
    再叠加配准残差，10cm 档必然低，20~30cm 档才是有效读数。
    （inner 口径下还要再加半墙厚系统差，见下"天花板效应"。）
- **matched corner mean dist（匹配角点平均距离）**：配上的角点之间平均差多少（0.144m）。
  可理解为"墙角平均偏 14.4cm"，其中约 10cm 量级来自上述像素量化+配准残差。

### 3. 墙线几何（boundary chamfer）

一句话：把预测的墙**边界线**和真实墙边界线比，平均离多远。

- **Chamfer 距离（倒角距离）**：在预测边界上均匀撒点，量每个点到 GT 边界的最近距离（再反过来量一遍），
  报 mean/median/P90（米）。逐房的 `chamfer mean` 列就是这个（如 `厨房 0.092m`）。它和角点距离
  互补：角点只看拐角，Chamfer 看**整条墙线**贴不贴，能抓到"角对上了但中间墙鼓出去"这类问题。

### 4. 门/窗位置（doors/windows，位置级）

一句话：门/窗/门洞画在了**哪里**，位置对不对。这是**唯一**的开口评估口径——早先基于
`openings_gt.json` 的"房间连通关系"评估已并入此处、统一改用 `doors_windows.json`。
仅当 GT 目录有 `doors_windows.json` 时评测，否则整段跳过。

预测的每个 `opening` 中心经配准 transform 映到 GT 系，与 GT 门窗逐个匹配：**同朝向**（横墙/竖墙一致）
**且中心距 ≤ `--dw_match_tol`（默认 0.6m）**，最近优先一对一。匹配上即算位置命中（TP），不管类型对不对。

- **P / R / F1**：位置级精确率/召回/调和平均。**匹配平均中心距**是命中门窗的平均定位误差。
- **mean width IoU（宽度 IoU）**：命中门窗的**宽度重叠程度**。门窗都落在墙上，把预测段与 GT 段
  投影到沿墙方向（横墙取 x、竖墙取 z）成两个 1D 区间，取**交/并**：1.0=起止完全一致，0=沿墙无重叠。
  同时反映**宽度对不对**和**位置偏不偏**。逐条明细还打印 `IoU x.xx (w pred a / gt b m)` 便于溯源。
- **by GT type（按类型分列召回）**：`door 6/9  window 0/8  opening 2/3`。**关键**：预测管线
  **不输出窗**（窗全在外墙、本版不产出），所以 window 恒为 0/N——单列出来，免得把它的漏检
  算到门/门洞头上、拖低总召回的判读。真正该看的是 door 和 opening 两列。
- **type agreement（类型一致）**：命中的门窗里，预测类型与 GT 类型也对上的数量（`6/8`）。
  预测的 `passage` 视作门洞（`opening`），`door` 视作门；两个 GT 门洞被判成门 → 位置命中但类型不符（标 `hit~`）。
- **明细**：`hit`/`hit~`（命中，`~`=类型不符）、`FP`（预测有、附近无 GT 门窗，如宽多房 passage）、
  `miss`（GT 有、预测漏，含全部窗）。

huizhongbeili-106 实测：`pred 9 / gt 20，matched 8，P 0.889 R 0.400 F1 0.552`，平均中心距 0.25m，
**mean width IoU 0.544**，`门 6/9、门洞 2/3、窗 0/8`，类型一致 6/8，唯一 FP 是跨 4 房的宽 passage。
R 偏低几乎全由"窗未产出"和上游房间合并造成，**门中心定位很准**（命中门平均 0.2m 级）；宽度 IoU 偏低
的多是**推拉门**（如客厅 pred 0.51 / gt 1.42m → IoU 0.31），说明模型对宽门洞的**宽度识别不足**。

`_eval_overlay.png` 会追加一个门/窗位置面板：**绿**=命中、**黄**=命中但类型不符（门洞判成门）、
**红实线**=虚报（FP）、**红虚线**=漏检门/门洞、**灰虚线**=漏检窗（预测本就不出窗）；命中的门窗还画
一条细连线连预测中心与 GT 中心、并在旁标注该处宽度 IoU，直观显示定位与宽度偏差。

### 5. 合并 GT 再评一轮（merged-GT re-evaluation）

一句话：如果我们**接受**预测把某几间合成一间，剩下的几何到底有多准？

第 1 组指标把"合并"整体记为未匹配，看不出合并房间本身画得好不好。所以在输出末尾自动
追加一轮：

- **判定**：某预测房把某 GT 房覆盖 **≥95%** 面积、且这样的 GT 房（连同与它 1:1 配对的那间）
  凑够 **两间以上**，就认定发生了合并。星河湾判出 3 组（映射关系逐行打印）：
  ```
  merge: p0   <- 卧室C + 阳台C
  merge: p12  <- 卧室A + 阳台A
  merge: p15  <- 客厅 + 餐厅 + 阳台B
  ```
  阈值故意取高：卫D（56%）/过道D（69%）是**分割错位**不是干净合并，不会被卷进来，
  仍以未匹配+诊断形式保留。**只合并 GT 侧，不合并预测侧**（分割问题不做对称处理）。
- **重评**：把每组 GT 房 union 成一间（名字用 `客厅+餐厅+阳台B`），配准**不重跑**，
  重算全部房间级 + 角点 + Chamfer + 总轮廓指标。
- **怎么读**（星河湾）：房间 17/17 对 17，P/R/F1 全部 0.737→**0.882**、mean IoU 0.849→**0.868**；
  合并大房本身非常准——`客厅+餐厅+阳台B` IoU **0.976**、Chamfer 仅 0.045m，
  `卧室C+阳台C` 0.941。剩余未匹配只剩 卫D/过道D 的分割错位。
  **第 1 轮与本轮的差距 = 房间合并对总分的代价**；本轮数字 = 墙体几何本身的质量。
- 两轮指标都写入 txt/json（json 键 `merged_eval`，含 `merge_groups` 映射）。
  inner 口径下合并 union 会保留房间间的真实墙缝（可能成 MultiPolygon），指标照常计算。

## 从页面 SVG 恢复缺失房间

星河湾的 `room_layout.json` 缺 3 房（卫生间A/衣帽间A/阳台C），但 realsee 页面的
户型图 tab 是一个 **每房一个 `<path>` 的内联 SVG**（毫米单位、y 向下），21 房齐全。

1. **抓 SVG**：页面是 JS 渲染的（直接 HTTP 只有空壳、CDN 无独立 JSON 端点），用
   **无头 Chrome + puppeteer-core**（复用系统 Chrome，无需下载浏览器）打开页面 →
   模拟点击"户型图"tab → 等出现 ≥5 path 的 svg → dump `outerHTML`，存为
   `{gt_dir}/floorplan.svg`。
2. **恢复**（方法概要，星河湾结果已固化在 `rooms_extra.json`）：
   - 解析 path（只允许 M/L/Z 纯多边形；弧形飘窗页面已离散成线段）；
   - 与 18 个已知 GT 房贪心质心匹配 + 逐轴最小二乘拟合 SVG→GT 变换
     （星河湾拟合 scale ≈0.001，即 mm→m，y 翻转；18/18 匹配，fit IoU 0.85）；
   - **内缩定标**：SVG 画的是**墙中线**、GT 是内表面，面积系统性偏大（中位数 +18%）。
     用已知房间反解 `buffer(-t)` 面积吻合的 t 取中位数（星河湾 0.096m ≈ 半墙厚），
     应用到恢复房间（卫生间A/衣帽间A 面积偏差降到 1–3%；阳台C 弧形窄条 -22% 可接受）；
   - 剩余 path 的**命名**来自门窗标注里的邻接关系：缺失房间各有一个已知邻居
     （卧室A↔卫生间A 等），距该邻居最近的剩余多边形取其名；
   - 产出 `rooms_extra.json`，并在 `floorplan.json` 顶层加 `rooms_extra` 字段
     （`local_path` 指向该文件）；`load_gt_rooms` 按此字段解析并自动合并。

## ⚠️ 天花板效应（仅 `--gt_geometry inner` 口径存在）

- `room_layout.json` 的 GT 多边形是房间**内表面**轮廓（相邻房间之间留有真实墙厚，
  如客厅↔阳台B 间 0.32m）；预测是**零厚度共享墙中线**。完美预测每边也差约半墙厚
  （6–16cm）：房间 IoU 上限 <1、角点距离下限 >0、预测面积系统性大于 GT 面积，
  拟合 scale 还会被额外压低 ~1%。
- **默认的中线口径（rooms_centerline.json）与预测同口径，无此系统差**——星河湾同一份
  预测，union IoU 从 0.900（inner）抬到 0.963（centerline），mean IoU 0.817→0.849，
  角点 @0.2m P 0.387→0.749。剩余误差主要是 256px 像素量化（~10cm/px）+ 配准残差 +
  真实预测误差。做绝对精度结论用中线口径；inner 口径仅用于与历史数字对比。

## 关键参数

| 参数 | 默认 | 说明 |
| --- | --- | --- |
| `--pred_units` | m | .ply 原始单位（MVS=米；Structured3D=毫米） |
| `--gt_geometry` | centerline | GT 口径：墙中线（与预测同口径）/ inner 内表面；无中线文件自动回退 inner |
